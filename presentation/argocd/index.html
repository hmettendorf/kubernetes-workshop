From Manual Deployments to Automated GitOps

🧭 What is GitOps?

GitOps is a paradigm for managing infrastructure and applications where Git is the single source of truth.

Core Principles:

  1. Declarative - System state described declaratively
  2. Versioned & Immutable - State stored in Git with full history
  3. Pulled Automatically - Agents pull desired state from Git
  4. Continuously Reconciled - Software agents ensure state matches declaration
💡 Key Insight: You don't push changes to production. You push to Git, and automation does the rest.

⚖️ Traditional vs GitOps

❌ Traditional (Push)

  • Manual kubectl apply
  • CI/CD pushes to cluster
  • Credentials in CI pipeline
  • No drift detection
  • Hard to audit

✅ GitOps (Pull)

  • Git commit triggers sync
  • Agent pulls from Git
  • No cluster credentials needed
  • Automatic drift detection
  • Full audit trail in Git
Result: Git becomes your deployment interface, rollback mechanism, and audit log!

📋 Declarative vs Imperative

Imperative (HOW):

kubectl create deployment nginx --image=nginx:1.21
kubectl scale deployment nginx --replicas=3
kubectl set image deployment/nginx nginx=nginx:1.22

Declarative (WHAT):

apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx
spec:
  replicas: 3
  template:
    spec:
      containers:
      - name: nginx
        image: nginx:1.22
💡 GitOps Advantage: Declarative configs can be versioned, reviewed, and rolled back easily.

🔄 Desired State & Reconciliation

Reconciliation Loop:

Git Repository (Desired State)
⬇️
Argo CD reads manifests
⬇️
Compares with Kubernetes (Actual State)
⬇️
Applies changes if drift detected
⬇️
Kubernetes Cluster (Synced State)

Key Benefits:

  • Self-healing - Manual changes are reverted
  • Drift detection - Know when reality differs from Git
  • Consistency - Same state across environments

📚 Git as Single Source of Truth

Everything in Git:

  • Application manifests (Deployments, Services, etc.)
  • Configuration (ConfigMaps, environment-specific values)
  • Infrastructure as Code (Helm charts, Kustomize overlays)
  • Policy definitions (NetworkPolicies, RBAC)
Benefits:
  • ✅ Complete audit trail (who changed what, when, why)
  • ✅ Easy rollback (git revert)
  • ✅ Collaboration via Pull Requests
  • ✅ Disaster recovery (recreate from Git)

🎯 Introducing Argo CD

Argo CD is a declarative, GitOps continuous delivery tool for Kubernetes.

Why Argo CD?

  • Kubernetes-Native - CRDs, controllers, familiar patterns
  • Declarative - Everything as code in Git
  • Automated Sync - Continuous reconciliation
  • Rich UI - Visual representation of apps
  • Multi-tenant - AppProjects for team isolation
  • Extensible - Supports Helm, Kustomize, and more
💡 Fun Fact: Argo CD has over 14k+ GitHub stars and is a CNCF graduated project!

🏗️ Argo CD Architecture

Core Components:

  • API Server - REST/gRPC API, Web UI, CLI
  • Repository Server - Clones Git repos, generates manifests
  • Application Controller - Monitors apps, compares desired vs actual state
  • Redis - Caching layer
  • Dex - Optional identity provider integration
User → API Server → Repository Server → Git
            ↓
            Application Controller → Kubernetes API

📦 Argo CD Applications

An Application is a CRD that represents a deployable unit.

Application Manifest:

apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: demo-app
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/myorg/myrepo
    targetRevision: HEAD
    path: k8s/production
  destination:
    server: https://kubernetes.default.svc
    namespace: production
  syncPolicy:
    automated:
      prune: true
      selfHeal: true

🎫 AppProjects

AppProjects provide logical grouping and multi-tenancy.

Project Capabilities:

  • Restrict which Git repos can be used
  • Limit destination clusters and namespaces
  • Define allowed/denied resource types
  • Set sync windows (maintenance windows)
  • Configure RBAC policies
💡 Use Case: Team A can only deploy to namespaces team-a-* from their repos, while Team B has their own restrictions.

🔄 Sync Strategies

Manual Sync:

  • Changes detected but not applied automatically
  • User must click "Sync" in UI or use CLI
  • Good for production environments requiring approval

Automated Sync:

  • automated: {} - Auto-sync on Git changes
  • prune: true - Delete resources removed from Git
  • selfHeal: true - Revert manual changes to cluster
syncPolicy:
  automated:
    prune: true
    selfHeal: true
  syncOptions:
  - CreateNamespace=true

🔍 Drift Detection

Drift occurs when cluster state differs from Git state.

Common Causes:

  • Manual kubectl changes
  • Other tools modifying resources
  • Operator or controller updates
  • Pod autoscaling (HPA)

Argo CD's Response:

  • Detection - Shows "OutOfSync" status
  • Diff View - Displays exact differences
  • Self-Heal - Auto-reverts if enabled
Result: You always know the true state of your applications!

⎈ Helm Integration

Argo CD has native Helm support - no Tiller needed!

Helm in Argo CD:

spec:
  source:
    repoURL: https://github.com/myorg/charts
    targetRevision: HEAD
    path: charts/demo-app
    helm:
      releaseName: demo
      valueFiles:
      - values-prod.yaml
      parameters:
      - name: image.tag
        value: "v1.2.3"
💡 Benefit: Use Helm for templating + Argo CD for GitOps. Best of both worlds!

🛠️ Kustomize Integration

Argo CD has built-in Kustomize support.

Kustomize in Argo CD:

spec:
  source:
    repoURL: https://github.com/myorg/k8s
    targetRevision: HEAD
    path: overlays/production
    kustomize:
      namePrefix: prod-
      commonLabels:
        environment: production
      images:
      - name: app
        newTag: v1.2.3

When to Use:

  • Environment-specific configurations
  • Patching base manifests
  • Simpler than Helm for basic needs

🎯 App of Apps Pattern

Deploy Applications that deploy Applications - recursive GitOps!

Use Cases:

  • Bootstrap entire environments
  • Manage multiple microservices
  • Team-specific application sets
Root App
├── Infrastructure App (monitoring, ingress)
├── Platform App (databases, queues)
└── Application Apps (microservices)
Result: Deploy entire platform with a single Application!

🚀 Environment Promotion

Strategy 1: Branch per Environment

  • dev branch → Dev cluster
  • staging branch → Staging cluster
  • main branch → Production cluster

Strategy 2: Directory per Environment

k8s/
├── base/
├── overlays/
│   ├── dev/
│   ├── staging/
│   └── production/
💡 Best Practice: Use Kustomize overlays or Helm values per environment in same repo.

💚 Health & Sync Status

Sync Status:

  • Synced - Git matches cluster
  • OutOfSync - Differences detected
  • Unknown - Unable to determine

Health Status:

  • Healthy - All resources running correctly
  • Progressing - Deployment in progress
  • Degraded - Some resources failing
  • Missing - Resources not found
  • Suspended - Paused resources (CronJobs)

🔐 RBAC in Argo CD

Control who can do what with Applications.

Example Policy:

p, role:developers, applications, get, default/*, allow
p, role:developers, applications, sync, default/*, allow
p, role:developers, applications, delete, default/*, deny

g, alice, role:developers

Actions:

  • get - View applications
  • sync - Trigger deployments
  • override - Bypass sync windows
  • delete - Remove applications

🔒 Handling Secrets

Problem: Don't store secrets in Git!

Solutions:

  • Sealed Secrets - Encrypt secrets for Git storage
  • External Secrets Operator - Sync from external vaults
  • Helm Secrets - SOPS encryption for Helm values
  • Vault - HashiCorp Vault integration
💡 Recommended: External Secrets Operator for production use.

🔧 Troubleshooting Tips

Common Issues:

  • OutOfSync forever
    • Check resource hooks, health checks
  • Sync fails silently
    • Review logs: kubectl logs -n argocd deployment/argocd-application-controller
  • Permission denied
    • Check AppProject restrictions
  • Helm chart issues
    • Test locally: helm template

Debug Commands:

argocd app get myapp
argocd app diff myapp
argocd app logs myapp

✨ GitOps Best Practices

  • ✅ One repo per team/project - Clear boundaries
  • ✅ Separate app code and config - Different repos for CI and CD
  • ✅ Use AppProjects - Enforce multi-tenancy
  • ✅ Enable prune and selfHeal - In dev/staging
  • ✅ Manual sync in production - With approval process
  • ✅ Meaningful commit messages - They're your deployment log
  • ✅ Use PR reviews - Production changes require review
  • ✅ Monitor sync status - Alert on sync failures

❌ GitOps Anti-patterns

  • ❌ Manual kubectl changes - Use Git or they'll be reverted
  • ❌ Secrets in Git - Use secret management tools
  • ❌ Single giant repo - Hard to manage, slow clones
  • ❌ No RBAC - Everyone can deploy everywhere
  • ❌ Ignoring drift - Defeats purpose of GitOps
  • ❌ Generated manifests in Git - Store templates, not generated YAML
  • ❌ Complex Helm values - Keep it simple and readable
